# Spec: mcp-roundtrip

**Created:** 2026-06-07
**Last updated:** 2026-06-07
**Status:** Draft

## Overview

MCP server (same backend process, Streamable HTTP at `/mcp`) for agents: publish docs
(create/update), read/list/search, pull annotations back to keep editing, and reply/resolve
comments. Agents authenticate with a personal API token. Closes the loop agent → annotator →
agent.

## Data Model

- **api_tokens**: `id`, `user_id`, `token_hash`, `name`, `last_used_at`,
  `created_at`, `revoked_at`.
- Reuse doc/version/annotation/comment; no separate model for content.

## Stories

### S-001: Authenticate agent with a personal API token (P0)

**Description:** As a user, I create (and revoke) an API token so an agent can call MCP under
my identity.
**Source:** docs/explore/mcp-roundtrip.md#decisions (item 1 agent auth, item 2 transport).

**Execution:**
- `depends_on:` none
- `parallel_safe:` false
- `files:` unknown (token issuance + MCP server mount at /mcp)
- `autonomous:` true
- `verify:` create token → agent calls MCP OK; revoke token → call rejected.

**Acceptance Scenarios:**

AS-001: Create a token and call MCP successfully
- **Given:** user creates an API token in settings
- **When:** agent calls an MCP tool with the token in the header to `/mcp` (Streamable HTTP)
- **Then:** the request executes under that user's identity
- **Data:** valid token

AS-002: Wrong/revoked token is rejected
- **Given:** a revoked token (or a wrong token string)
- **When:** agent calls MCP with that token
- **Then:** rejected; agent gets a clear auth error
- **Data:** token revoked

### S-002: Publish via MCP — create & update (P0)

**Description:** As an agent, I create a new doc (receiving docId/slug) and update an existing
doc (appending a new version).
**Source:** docs/explore/mcp-roundtrip.md#decisions (item 3 publish tools).

**Execution:**
- `depends_on:` S-001
- `parallel_safe:` false
- `files:` unknown (map onto the identity model of render-publish/versioning)
- `autonomous:` true
- `verify:` create_document → returns docId/slug + v1; update_document(docId) → v2 appears.

**Acceptance Scenarios:**

AS-003: create_document creates doc + slug + version 1
- **Given:** agent is authenticated
- **When:** calls `create_document(content, format, title?, projectId?)`
- **Then:** creates a doc with an immutable slug + version 1, general_access = **restricted**
  (token-owner only), placed in the token-owner's default project if projectId is missing;
  returns `{ docId, slug, url }`
- **Data:** HTML content + title "Payment Spec", no projectId passed

AS-004: update_document appends a new version
- **Given:** a doc already exists (docId), agent has editor+ rights on the doc
- **When:** calls `update_document(docId, content)`
- **Then:** appends a new version (no overwrite); triggers re-anchor (versioning/annotation)
- **Data:** docId + edited content

AS-005: update_document with nonexistent docId / no permission is rejected
- **Given:** agent calls update with a nonexistent docId or without editor rights
- **When:** executes
- **Then:** rejected; suggests create_document if a new doc is wanted
- **Data:** unknown docId / token-owner has viewer rights only

### S-003: Read via MCP — list / read / search (P1)

**Description:** As an agent, I list, read, and search docs within the token-owner's
permission scope.
**Source:** docs/explore/mcp-roundtrip.md#decisions (item 3 read).

**Execution:**
- `depends_on:` S-001
- `parallel_safe:` false
- `files:` unknown
- `autonomous:` true

**Acceptance Scenarios:**

AS-006: list/read/search within the permission scope
- **Given:** token-owner has rights on some docs
- **When:** agent calls `list_documents` / `read_document(idOrSlug)` / `search_documents(query)`
- **Then:** returns only docs the token-owner has rights to; docs outside the scope do not appear
- **Data:** query matches docs both inside and outside the scope

### S-004: Pull annotations (P0)

**Description:** As an agent, I pull a doc's annotations to locate them and edit them in my
own source.
**Source:** docs/explore/mcp-roundtrip.md#decisions (item 3 pull annotations).

**Execution:**
- `depends_on:` S-001
- `parallel_safe:` false
- `files:` unknown
- `autonomous:` true

**Acceptance Scenarios:**

AS-007: pull_annotations returns enough context to locate
- **Given:** doc has annotations (including resolved, orphaned) + a suggestion
- **When:** agent calls `pull_annotations(docId)`
- **Then:** returns annotation + comment thread + `status` (resolved/unresolved,
  is_orphaned) + suggestion (type delete/replace + content) + anchor (block_id +
  text_snippet) for the agent to locate in source
- **Data:** doc has 1 comment + 1 replace suggestion + 1 orphaned

AS-008: pull only fetches annotations new since last time (since-cursor)
- **Given:** agent pulled earlier with a cursor
- **When:** calls pull with that cursor
- **Then:** returns only annotations/comments new or changed since the cursor, no repeats of already-handled ones
- **Data:** previous cursor + 2 new comments

### S-005: Write back — reply & resolve (P1)

**Description:** As an agent, after handling feedback I reply and mark it resolved.
**Source:** docs/explore/mcp-roundtrip.md#decisions (item 3 write back).

**Execution:**
- `depends_on:` S-001
- `parallel_safe:` false
- `files:` unknown
- `autonomous:` true

**Acceptance Scenarios:**

AS-009: reply_comment and resolve_comment
- **Given:** an annotation has a comment, token-owner has comment rights on the doc
- **When:** agent calls `reply_comment(commentId, body)` then `resolve_comment(annotationId)`
- **Then:** the reply is added to the thread (flat); the annotation becomes resolved
- **Data:** reply "changed to 48h in v2" + resolve

AS-010: token hashed + pull authorize per-doc [harden H4]
- **Given:** a token; token-owner has rights on doc A, not on doc B (restricted)
- **When:** agent calls `pull_annotations(B)`; and the token storage is observed
- **Then:** pull(B) rejected (authorize per-doc by owner); token stored as a hash,
  no plaintext exposed; revoking the token rejects all subsequent calls
- **Data:** token-owner ∉ rights on doc B

## Constraints & Invariants

- C-001: Token = the owning user's rights; revoked/wrong token → rejected. (AS-001, AS-002)
- C-002: create → immutable slug + version 1 + returns docId; update → append version
  (no overwrite), requires editor+ rights. (AS-003, AS-004, AS-005)
- C-003: list/read/search return only docs the token-owner has rights to. (AS-006)
- C-004: pull_annotations returns status (resolved/orphaned) + suggestion + anchor
  (block_id + text_snippet) enough for the agent to locate. (AS-007)
- C-005: Transport Streamable HTTP at `/mcp`. (AS-001)
- C-006: Visibility is NOT set via MCP in v0; docs an agent creates default to **restricted**
  (token-owner only), placed in the token-owner's default project if projectId is missing. (AS-003)
- C-007: MCP has per-token rate-limiting. (AS-002)
- C-008 [harden H4]: api_tokens stored HASHED (no plaintext), shown once at creation,
  revoke individually, with `last_used_at` + optional expiry, minimum scope read vs write;
  pull/read/write authorize per-doc/per-annotation like the web path, by the owner token. (AS-010)

## Linked Fields

- **docId/slug + version** — produced by mcp:S-002 (create/update), matches the identity
  model of `render-publish`/`versioning-diff`. ✔ same doc+version model.
- **anchor + suggestion + status** — produced by `annotation-core` (Data Model).
  Consumed by mcp:S-004 (AS-007) to return to the agent. ✔ pull reads the correct block-anchor model.
- **user's default project** — produced by `workspace-project` (each account
  auto-creates a default project). Consumed by mcp:S-002 (AS-003) as the container for docs the agent
  creates when projectId is missing. ✔ workspace-project creates a default project per-user (C-009 in
  that spec).

## What Already Exists

### System Impact & Technical Risks

- Repo greenfield. MCP server in the same Elysia process, using `@modelcontextprotocol/sdk`.
- Cross-spec: create/update = doc+version (`render-publish`/`versioning-diff`); pull/
  reply/resolve use the `annotation-core` model; token tied to user (`auth`); endpoint `/mcp`
  + rate-limit (`self-host`).
- Reuse: no separate content model — MCP is an API surface over existing models.

## Not in Scope

- OAuth 2.1 agent auth (better-auth provider) → v0.5 (v0 uses API token).
- upload_asset / upload_zip / publish_with_assets (rewrite img→asset) → ships with zip, v0.5.
- set_access / unpublish via MCP → v0.5.
- unorphan / relocate annotation via MCP → v0.5 (v0 done by a person on the UI).
- moderate / delete_document / reactions via MCP → v0.5+.
- n8n integration → v2.
- stdio transport → not doing.

## Clarifications — 2026-06-07

- **API token instead of OAuth in v0:** the agent runs local/CI, a header token is the
  simplest self-host approach.
- **Streamable HTTP:** the instance is already a server; a remote/CI agent needs HTTP; stdio is limited.
- **Full round-trip (including reply/resolve):** the "agent edits then marks as handled" loop is what
  truly closes it.
- **create returns slug + update appends version:** matches the identity model locked in
  render-publish/versioning.
- **since-cursor for pull:** so the agent doesn't reprocess already-handled annotations.
