# Spec: Annotation Hover Peek + Click-to-Pin Card

**Created:** 2026-06-26
**Last updated:** 2026-06-26
**Status:** Draft (hardened via /mf-challenge 2026-06-26)

## Overview

In the doc viewer, a reviewer must currently look to the right-hand rail to read or act on an
annotation. This feature surfaces the annotation in place: hovering an in-doc marker shows a
read-only "peek" card after a short dwell, and clicking the marker pins a full, interactive
thread card in a floating popover anchored at the marker. The rail stays as-is and parallel.
All data is already loaded client-side, so no API or schema change is involved.
Source: `docs/explore/annotation-hover-card.md#overview`.

## Data Model

None. No new entities, fields, or migration. The cards render from the annotation/comment data
already fetched for the rail (`ViewerAnnotation` / `AnnotationComment` in
`apps/web/src/features/viewer/services/client.ts`).
Source: `docs/explore/annotation-hover-card.md#data-impact`.

## Stories

### S-001: Hover peek on a marker (P0)

**Description:** As a reviewer reading a markdown doc, when I rest the cursor on an annotation
marker I see a small read-only card summarizing that annotation, so I can glance at it without
moving to the rail.
**Source:** `docs/explore/annotation-hover-card.md#happy-path` (steps 2-3) + `#business-rules` (dwell, read-mode-only).
**Applies Constraints:** C-001, C-003, C-008

**Execution:**
- `depends_on:` none
- `parallel_safe:` false
- `files:` `apps/web/src/features/viewer/components/annotation-marks.tsx`, `apps/web/src/features/viewer/components/annotation-peek-card.tsx` (new), `apps/web/src/features/viewer/hooks/use-hover-pin.ts` (new), `apps/web/src/features/viewer/components/viewer-screen.tsx`
- `autonomous:` true
- `verify:` `cd apps/web && bun test annotation-peek-card`
- `contract:` (a) hover is detected on the shared doc-pane listener via `mouseover`/`mouseout` with a `relatedTarget` check — NOT per-mark `mouseenter`/`mouseleave` (which do not bubble to a delegated listener); a dwell timer (~200ms) is started on entering a mark and CANCELLED when the cursor moves to a non-mark target or a different annotation; the timer callback re-checks the cursor is still over the same mark before showing. (b) The peek/pin anchor rect is the HOVERED/CLICKED mark's own `getBoundingClientRect()` (`e.target.closest(MARK_SELECTOR)`), never `placed.el` (one annotation = N marks for a `multi_range`); moving between marks that share the same `data-anno` is NOT a leave (coalesce by id). (c) Suppression (C-001) is gated on TEXT-SELECTION state, not tool identity: the delegated handler shows the peek only when there is no in-progress / non-empty text selection (e.g. `window.getSelection()` is collapsed/empty). The selection state must reach the listener via a ref or the effect's deps so the delegated handler never reads a stale value. Do NOT gate on `activeTool`.

**Acceptance Scenarios:**

AS-001: Dwell on a marker shows the peek card
- **Given:** a markdown doc in read mode (no annotation-creation tool active) with an annotation anchored to the phrase "tokens before the refresh"
- **When:** the reviewer rests the cursor over that marker for the dwell interval (~200ms)
- **Then:** a read-only peek card appears anchored above the marker
- **Data:** annotation by Jane Smith, created 2h ago, 1 root comment + 3 replies, status unresolved
- **Setup:** viewer loaded at `/d/spec-abc`, cursor not previously over the marker

AS-002: Moving the cursor off the marker hides the peek
- **Given:** the peek card from AS-001 is showing
- **When:** the cursor leaves the marker (and does not enter the peek)
- **Then:** the peek card disappears
- **Data:** same annotation as AS-001

AS-003: An in-progress text selection suppresses the peek (error/edge path)
- **Given:** the reviewer has an active (non-collapsed) text selection in the doc — they are mid-annotate — with any tool selected
- **When:** the cursor rests over an existing marker for longer than the dwell interval
- **Then:** no peek card appears (the annotate-create flow owns the gesture); with NO active selection, the peek appears normally regardless of which tool is selected
- **Data:** a non-empty selection spanning part of a paragraph that also carries a marker; Markup tool selected

AS-004: Peek content is the condensed summary, read-only
- **Given:** a marker for an annotation with a long thread
- **When:** the peek appears
- **Then:** it shows author + relative time + a status indicator + a type chip when applicable + the quoted phrase (clamped) + the root comment (clamped) + a remaining-reply count, and shows NO Reply/Resolve/Accept controls
- **Data:** redline annotation by Jane Smith, root + 3 replies → the peek shows "3 replies", a Delete type chip, and no action bar

AS-028: The peek status indicator matches the annotation's own colour
- **Given:** a marker whose annotation carries a type/label colour (the SAME colour as its in-doc highlight + its rail row)
- **When:** the peek appears for it
- **Then:** the peek's status indicator is rendered in THAT colour — comment amber, any label (incl. like) gold, redline red, otherwise the default teal accent — so it is recognizably the same colour as the highlight being hovered; never a generic accent/success dot that ignores the annotation's colour
- **Data:** a plain comment → amber; a label/like annotation → gold; a redline → red; a bare comment with no type → teal

AS-029: A resolved annotation's peek indicator turns green
- **Given:** a marker whose annotation is resolved (its in-doc highlight shows the resolved/green tint)
- **When:** the peek appears for it
- **Then:** the peek's status indicator is green (success), mirroring the mark's own resolved tint — resolved state takes precedence over the type hue, exactly as the mark renders it
- **Data:** a resolved comment annotation → green indicator (not amber)

AS-005: A marker near the viewport edge keeps the whole peek on-screen
- **Given:** an annotation marker near the top edge of the viewport
- **When:** the peek would not fit above the marker
- **Then:** it flips to below the marker and is clamped horizontally so the whole card stays visible
- **Data:** marker whose top is within the peek height of the viewport top

AS-022: A cross-block annotation anchors the peek to the hovered segment, not the first
- **Given:** a `multi_range` annotation that draws two markers far apart — one in a heading, one in a paragraph well below it — both carrying the same annotation id
- **When:** the reviewer hovers the lower (paragraph) marker
- **Then:** the peek anchors to the lower marker's own rect (over the cursor), not to the first/heading marker
- **Data:** annotation whose anchor has two segments in different blocks ~300px apart

AS-025: Sweeping the cursor across markers does not leave an orphan peek
- **Given:** read mode with several markers along one line of text
- **When:** the reviewer moves the cursor quickly across them without resting (each under the dwell interval), then off all marks onto plain text
- **Then:** no peek card is shown — every pending dwell timer is cancelled when the cursor leaves a mark before the interval elapses
- **Data:** three adjacent annotations; cursor traverses all three in under the dwell interval

### S-002: Click a marker to pin the full thread card (P0)

**Description:** As a reviewer, when I click an annotation marker I get the full annotation
thread pinned in a floating card at the marker, where I can reply, resolve, and decide — without
the rail losing its place.
**Source:** `docs/explore/annotation-hover-card.md#happy-path` (steps 4-5) + `#decision-rationale` (pinned = full ThreadCard; rail parallel).
**Applies Constraints:** C-001, C-002, C-004, C-005, C-007

**Execution:**
- `depends_on:` S-001
- `parallel_safe:` false
- `files:` `apps/web/src/features/viewer/components/annotation-marks.tsx`, `apps/web/src/features/viewer/components/pinned-card-popover.tsx` (new), `apps/web/src/features/viewer/hooks/use-hover-pin.ts`, `apps/web/src/features/viewer/components/viewer-screen.tsx`, `apps/web/src/features/viewer/lib/place-popover.ts` (reuse)
- `autonomous:` true
- `verify:` `cd apps/web && bun test pinned-card-popover`
- `contract:` (a) the open card's state is a `pinnedId` DISTINCT from `focusedId` (C-004): pinning sets both; rail focus (AS-007) reads `focusedId`; the same-marker toggle (AS-011) reads `pinnedId`. (b) The outside-click dismiss treats BOTH the popover AND the pinned marker as "inside" (the marker that opened the card must not auto-dismiss it via the capture-phase `mousedown`); a re-click on the pinned marker resolves to exactly ONE toggle-close. (c) Escape is layered: when the card's reply composer is open, Escape cancels the reply only (stops propagation); Escape with no inner editor open closes the card. (d) The pin auto-closes when `pinnedId` no longer resolves to an annotation in the loaded list (deleted, dismissed, orphaned-out, refetch-dropped, or a redline rejected → its mark removed).

**Acceptance Scenarios:**

AS-006: Clicking a marker pins the full thread card
- **Given:** a markdown doc in read mode with an annotation marker
- **When:** the reviewer clicks the marker
- **Then:** a floating popover opens at the marker hosting the full thread card — full comment thread plus the same Reply / Resolve / Reopen / Accept / Reject affordances the rail card offers, positioned preferring below the marker and flipped/clamped to stay on-screen
- **Data:** unresolved comment annotation, root + 2 replies, current user is commenter
- **Setup:** no card currently pinned

AS-007: Pinning highlights the matching rail card while the rail stays visible
- **Given:** the rail is visible listing all annotations
- **When:** the reviewer pins a card by clicking its marker
- **Then:** the marker shows the focused emphasis AND the matching rail card shows the focused/current state, and the rail remains fully visible and usable
- **Data:** doc with 12 annotations; clicking marker #5 focuses rail card #5

AS-008: Click outside dismisses the pinned card
- **Given:** a card is pinned
- **When:** the reviewer clicks elsewhere in the doc — not on the card AND not on the card's own pinned marker
- **Then:** the pinned card closes
- **Data:** click target is plain doc text away from the marker

AS-009: Escape dismisses the pinned card
- **Given:** a card is pinned
- **When:** the reviewer presses Escape
- **Then:** the pinned card closes

AS-010: The close control dismisses the pinned card
- **Given:** a card is pinned
- **When:** the reviewer activates the card's close (✕) control
- **Then:** the pinned card closes

AS-011: Clicking the same marker again toggles the card closed
- **Given:** a card is pinned for marker M
- **When:** the reviewer clicks marker M again
- **Then:** the pinned card closes exactly once — the re-click does NOT both toggle-close and outside-dismiss (the marker is excluded from the outside-dismiss test)
- **Data:** same marker, no other card opened in between

AS-012: Pinning a second marker closes the first (at most one pinned)
- **Given:** a card is pinned for marker A
- **When:** the reviewer clicks a different marker B
- **Then:** the card for A closes and a card for B is pinned — never two cards at once
- **Data:** two distinct annotations A and B in the same doc

AS-013: An in-progress text selection suppresses pinning (error/edge path)
- **Given:** the reviewer has an active (non-collapsed) text selection in the doc — they are mid-annotate — with any tool selected
- **When:** the reviewer's gesture ends on an existing marker
- **Then:** no card is pinned; the annotate-create flow owns the gesture; with NO active selection, clicking a marker pins normally regardless of which tool is selected
- **Data:** a drag-selection ending on a marker, Comment tool selected

AS-014: Pinned-card actions follow the reviewer's role
- **Given:** the reviewer's effective role on the doc is viewer (read-only)
- **When:** a card is pinned
- **Then:** the card shows the thread read-only with no Reply / Resolve / Accept / Reject controls
- **Data:** viewer-role session; annotation with an open thread

AS-021: A pinned card auto-closes when its marker scrolls out of view
- **Given:** a card is pinned for a marker in a markdown doc
- **When:** the reviewer scrolls the doc pane until that marker leaves the viewport
- **Then:** the pinned card closes (driven by a throttled doc-pane scroll/resize listener re-reading the mark rect via `isRectOutOfViewport`)
- **Data:** long markdown doc; pinned marker scrolled past the top edge

AS-023: Focusing another thread via the rail while a card is pinned closes the pin
- **Given:** a card is pinned for marker A
- **When:** the reviewer clicks a DIFFERENT thread card B in the rail
- **Then:** the pin for A closes and B becomes the focused thread (the pin does not linger on A while the rail shows B) — at most one pinned card (C-002)
- **Data:** doc with annotations A and B; pin opened on A, then rail card B clicked

AS-024: Escape cancels an open reply before closing the card
- **Given:** a card is pinned and its reply composer is open with typed text
- **When:** the reviewer presses Escape
- **Then:** the reply composer cancels (the card stays open and the draft input clears); a second Escape (no composer open) closes the card
- **Data:** reply draft "needs a rethink"; two Escape presses

AS-026: Deleting the annotation from inside the pinned card closes the pin
- **Given:** a card is pinned for an annotation the reviewer authored, with Delete available
- **When:** the reviewer deletes the annotation from inside the pinned card
- **Then:** the pinned card closes (it does not linger over a now-removed annotation), consistent with the undo/restore flow the rail already owns
- **Data:** own annotation; optimistic remove in flight

### S-003: Hover peek and pin on HTML docs (P1)

**Description:** As a reviewer of an HTML doc (rendered in the sandboxed iframe), I get the same
hover peek and click-to-pin behavior as on markdown docs.
**Source:** `docs/explore/annotation-hover-card.md#unhappy-paths` (HTML iframe) + `#technical-risks` (bridge relay).
**Applies Constraints:** C-001, C-002, C-003, C-004, C-005, C-006

**Execution:**
- `depends_on:` S-001, S-002
- `parallel_safe:` false
- `files:` `apps/backend/src/annotation/sandbox-bridge.ts` (the in-iframe script — the ONLY code that can observe iframe pointer events), `apps/web/src/features/viewer/lib/bridge.ts` (parent relay handler + Zod parse), `apps/web/src/features/viewer/components/html-sandbox-frame.tsx`, `apps/web/src/features/viewer/components/viewer-screen.tsx`
- `autonomous:` checkpoint
- `verify:` `cd apps/backend && bun test sandbox-bridge` AND `cd apps/web && bun test bridge`
- `contract:` an in-iframe `mark-click` relay ALREADY EXISTS — `sandbox-bridge.ts` posts `{ type: "mark-click", annotationId }` (field is `annotationId`, NOT `annoId`, and it carries NO rect today), and the parent already routes it (`onMarkClick` → `viewer-screen` focus + rail-drawer open). So this story EXTENDS the existing relay rather than inventing one: (a) add the mark's `rect` to the `mark-click` payload and add new `mark-enter` / `mark-leave` (peek) messages, all rAF-throttled like the existing `selection-rect` relay; (b) RE-ROUTE the existing `onHtmlMarkClick` to drive the pin (reconciled with S-004 so mobile opens the sheet, not the old drawer); (c) for AS-021 on HTML add a `mark-rect`-on-scroll relay (the parent cannot see iframe scroll) so the pinned card can close/reposition. Cross-app change (`apps/backend` + `apps/web`).

**Acceptance Scenarios:**

AS-015: Hovering a marker inside an HTML doc shows the peek outside the sandbox
- **Given:** an HTML doc rendered in the sandboxed iframe, with an annotation marker inside it, in read mode
- **When:** the reviewer rests the cursor over the in-iframe marker for the dwell interval
- **Then:** the same peek card appears, rendered in the parent document positioned over the marker (its position translated by the iframe's offset)
- **Data:** HTML doc, annotation by Bob Lee, root + 1 reply

AS-016: Clicking a marker inside an HTML doc pins the card outside the sandbox
- **Given:** an HTML doc with an in-iframe marker, read mode, no card pinned
- **When:** the reviewer clicks the in-iframe marker
- **Then:** the full thread card pins in the parent document at the marker's translated position, with the same focus/dismiss behavior as on markdown
- **Data:** HTML doc, unresolved comment annotation

AS-017: HTML-doc card content renders as inert text (security path)
- **Given:** an annotation on an HTML doc whose quoted phrase and a comment body contain HTML-like markup (e.g. `<img onerror=...>`)
- **When:** the peek or pinned card renders that content
- **Then:** the markup is shown as literal text and never interpreted — the card content comes only from the trusted, already-loaded annotation data, never from anything in the relayed message
- **Data:** comment body = `<script>alert(1)</script>`, quote = `<b>bold</b>`

AS-027: A relayed mark event for an unknown or forged id opens nothing (security path)
- **Given:** an HTML doc whose body script posts a `mark-click` (or hover) over the port carrying an annotation id that is NOT in the viewer's loaded, role-filtered annotation set — or a malformed/oversized rect
- **When:** the parent receives the relayed message
- **Then:** the message is schema-validated; an id not found in the loaded set is a no-op (no card opens); a non-finite/negative/over-viewport rect is rejected and the card position is clamped to the viewport
- **Data:** forged id `"ann-not-loaded"`; rect `{ width: 1e9, height: 1e9 }`

## Constraints & Invariants

C-001: Hover peek and click-to-pin are gated on TEXT-SELECTION state, NOT on which tool is active. The viewer always has a tool selected (default Markup; there is no none/read tool), so tool-identity cannot gate. Peek (hover) and pin (click on a marker) are ACTIVE whenever there is no in-progress / non-empty text selection — regardless of which tool (Markup / Comment / Redline / Label) is selected. They are SUPPRESSED only while the user is actively selecting text (a non-collapsed selection exists, selectStart..selectEnd), so the annotate-create flow owns that gesture and is never overridden.
  - scope: S-001, S-002, S-003
  - surfaces: peek, pin
  - coverage: peek → AS-003, pin → AS-013
C-002: At most one card is pinned at any time — opening a new pin (or bottom sheet) closes the previous one. (AS-012)
C-003: The quoted phrase and comment bodies are untrusted and render as literal text only, never as interpreted HTML/markup — this holds for the new peek/pin/sheet surfaces, including the HTML-doc case where the card renders outside the sandbox. (AS-017)
C-004: Pinning a card focuses the matching rail card and vice-versa (they share `focusedId`), BUT the pin's open/closed state is a SEPARATE `pinnedId` — the same-marker toggle (AS-011) and the auto-close-on-unresolved rule key off `pinnedId`, never off `focusedId` (which stays set for the rail highlight). At most one `pinnedId` at a time. (AS-007, AS-011)
C-005: Actions inside the pinned card / bottom sheet reuse the existing per-role gating of the rail thread card (commenter+ to reply/resolve, owner-only accept/reject, author/owner to delete); no new permission is introduced and the backend re-authorizes every write. (AS-014, AS-020)
C-006: Every message relayed from the sandboxed iframe to the parent (`mark-click` / `mark-enter` / `mark-leave` / `mark-rect`) is UNTRUSTED input: it is schema-validated (Zod) at the parent boundary; its `annotationId` is looked up in the loaded, role-filtered annotation set and a miss is a no-op (no card opens); its `rect` is rejected when non-finite/negative/over-viewport and the card position is clamped to the viewport. A relayed open is PRESENTATION-only over server-trusted content — card actions always re-derive role/gating from the parent session, never from the message. (AS-027)
  - scope: S-003
  - surfaces: mark-click, mark-enter
  - coverage: mark-click → AS-027, mark-enter → AS-027
C-007: The pinned card and the matching rail card are two independent optimistic views of one thread; a reply/resolve made in one is reconciled into the other on the next refetch (transient divergence is accepted, per the async model). The rail card MAY be de-emphasized while its thread is pinned to reduce the duplicate-surface confusion. (AS-006, AS-007)
C-008: The peek/pin anchors to the rect of the actual hovered/clicked mark element, and marks sharing one `data-anno` (a `multi_range` annotation's N marks) are treated as one annotation for hover purposes — moving between them is not a leave. (AS-022, AS-025)

### System Impact & Technical Risks

- `useAnnotationMarks` / `viewer-screen.tsx`: gain hover/dwell + pin state; the current
  click→`onFocusAnno` is preserved (in read mode it now ALSO opens the pin), and `focusedId`
  is reused as the single active-annotation source (C-004).
- **HTML-doc iframe bridge (primary risk):** an in-iframe `mark-click` relay ALREADY exists and
  is wired — `apps/backend/src/annotation/sandbox-bridge.ts` posts `{ type: "mark-click",
  annotationId }` (no rect) over the dedicated MessageChannel port; `bridge.ts` `onMarkClick`
  forwards it and `viewer-screen` already focuses the rail + opens the drawer. S-003 EXTENDS this:
  add `rect` to `mark-click`, add `mark-enter`/`mark-leave` for the peek, add a `mark-rect`-on-scroll
  relay for AS-021, and RE-ROUTE `onHtmlMarkClick` to pin (reconciled with S-004's mobile sheet).
  The in-iframe half lives in `apps/backend` (not just FE `bridge.ts`); the relay is rAF-throttled
  like the existing `selection-rect` message; the relayed payload is untrusted (C-006). Cross-app +
  trust-boundary change → S-003 is `autonomous: checkpoint`.
- **Dual optimistic state (C-007):** the pinned card and the rail both mount `ThreadCard`, each with
  its own `optimisticReplies`/`optimisticResolved` `useState`. A reply/resolve in one shows in the
  other only after refetch — accepted transient divergence; consider de-emphasizing the rail card
  while pinned.
- **Stale-closure hazard (C-001):** `activeTool` is `viewer-screen` state while the click/hover
  delegation lives in `useAnnotationMarks` over the raw DOM element — the tool flag must reach the
  listener via a ref or be in the effect deps, else the suppression reads a stale value.
- happy-dom has no layout (`getBoundingClientRect` → 0): keep positioning math pure +
  unit-tested with synthetic rects (mirror `place-popover.ts`); live placement is
  Playwright/manual.
- No existing bottom-sheet primitive — S-004 builds one honoring DESIGN.md tap targets (≥44px)
  and the teal accent.

## Not in Scope

- **Image-region hover/pin** — image-region annotations do not exist yet (`ImageViewer` is
  display + zoom only; image annotation is deferred to `annotation-core` G11/suggest-image).
  There is no image marker to hover, so this waits until image-region annotations ship.
  Source: `docs/explore/annotation-hover-card.md#decision-rationale` + Phase-0 finding `image-viewer.tsx:7`.
- **Interactive-on-hover card** (moving the cursor into the hover card to act) — deliberately
  rejected in favor of peek + click-to-pin to avoid the "cursor can't reach the card" problem.
- **A new active-annotation concept** separate from `focusedId` — reuse `focusedId`.
- **Replacing/hiding the rail** when a card is pinned — rail stays parallel.
- **Retiring the existing mobile rail-drawer + CommentFab** — kept; the bottom sheet is additive.
- **Real-time push of new replies** into an open card — async model; refetch reconciles as today.
- **Anchor editing / re-attach from the card** — rail-only, unchanged.

## Clarifications — 2026-06-26 (S1 resolution from /mf-build)

- **Peek/pin gating is selection-based, NOT tool-based.** The viewer always has a tool selected
  (default Markup; no none/read tool), so tool-identity cannot gate the feature — gating on Markup
  would make peek/pin never appear. The gate is the presence of an in-progress / non-empty text
  selection: no selection → peek on hover + pin on marker click, regardless of the active tool;
  an active selection (the user is mid-annotate) → suppressed so the annotate-create flow wins.
  Rebased C-001, AS-003, AS-013, and the S-001 contract note (c). S-001 was already built gating on
  tool — its AS-003 + suppression logic are reopened for selection-based rework on /mf-build resume.

## Clarifications — 2026-06-26 (peek-status colour, dogfood feedback)

- The peek's status indicator must be the SAME colour the annotation shows everywhere else (its
  in-doc highlight mark + its rail row): per-type/label hue — comment amber, any label/like gold,
  redline red, default teal — and green when resolved (mirroring the mark's resolved tint,
  precedence over hue). The original binary teal/green dot ignored the annotation's colour and read
  as meaningless. Encoded as AS-028 (hue) + AS-029 (resolved precedence) under S-001.
